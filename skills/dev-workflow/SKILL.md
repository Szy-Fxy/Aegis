# 结构化需求开发工作流 v3.0

## 概述

当用户提交开发需求时，不要直接写代码。AI 先判断需求级别（L1/L2/L3），再按对应流程推进。

核心原则：**需求分级、技术栈匹配、人审关键节点、迭代可退回、写开发日志**。

---

## 规则文件加载协议

### 加载顺序

```
1. Aegis/rules/global.md                      ← 全局通用准则（必须加载）
2. Aegis/rules/TechStack/{匹配的技术栈}.md     ← 按需加载
3. 项目根目录 README.md / CLAUDE.md            ← 项目入口，可能含覆盖规则
```

### 技术栈自动发现

AI 根据用户需求的关键词自动匹配技术栈：

| 关键词 | 匹配文件 |
|--------|----------|
| Unity、C#、MonoBehaviour、游戏 | `TechStack/unity.md` |
| Python、Django、FastAPI、pytest、脚本 | `TechStack/python.md` |
| TypeScript、React、Vue、Node、Express、全栈、Web | `TechStack/typescript.md` |
| Unreal、UE5、Blueprint、Actor | `TechStack/unreal.md` |
| C++、CMake、vcpkg、算法（非 UE） | `TechStack/cpp.md` |

### 技术栈文件不存在时

如果匹配不到已有技术栈文件：

```
1. [搜索代理] 派子代理上网搜索：
   - 命名规范和代码风格惯例
   - 主流包管理器 / 构建工具
   - 社区最常用的第三方库（按下载量排序）
   - 常见陷阱和反模式
   - 项目目录结构的最佳实践

2. [审查代理] 派第二个子代理做「唱反调审查」：
   - 排除「一次性方案」（仅适用特定项目）
   - 排除过时库/工具（2年以上未更新）
   - 排除社区争议大的
   - 确认是「技术栈通用知识」而非「特定项目偏方」
   - 提供开源仓库链接、官方文档链接作为佐证

3. [暂存资料] 审查通过的资料存入：
   Aegis/rules/TempData/{技术栈名}_research_{YYYY-MM-DD}.md
   格式见 conventions/naming-and-formats.md

4. [生成文件] 基于审查过的资料生成：
   Aegis/rules/TechStack/{技术栈名}.md
   格式参照已有文件（如 python.md）

5. [清理] 确认无误后，可删除 TempData 研究文件或保留作为追溯
```

### TempData 文件夹用途

存放开发过程中的临时参考资料。格式和提炼标准见 `conventions/naming-and-formats.md`。

**核心规则**：
- TempData 中的资料必须经过审查代理验证
- 技术栈文件定位为「引子」（广而泛），不要求专精
- 当本地 TechStack 不够用时 → 网上查 → 放 TempData → 用完后提炼精华合并回 TechStack
- 提炼标准：被 ≥2 个需求引用 + 通用知识 + 可验证来源 + 提炼后不超过 TechStack 的 30%

## 进程恢复（Process State）

每次对话开始时的标准动作：

```
1. 读取 Aegis/rules/DevLogs/ 下最新日期的 DevLog
2. 如果信息不足（不知道上次走到哪了），依序读取上一份
3. 根据 DevLog 中的「当前进度」恢复上下文
4. 告知用户：「上次您在 {需求名} 的 {阶段}，是否继续？」
```

DevLog 文件名格式：`{项目名}_P{两位数字}_{功能描述}_{YYYY-MM-DD}.md`
时间精度到天。如果同一天有多份，按文件名排序取最新。

## 需求输入引导

### 引导对话

当用户需求模糊时，AI 主动引导：

```
我理解你想 [做某件事]。在开始之前，帮我确认几个关键信息：

1. 这个功能是解决什么问题的？（业务背景）
2. 完成后谁会用？怎么用？（用户场景）
3. 有什么绝对不能变的东西吗？（约束条件）
4. 你已经有具体方案了，还是需要我先想几个方案供你选？

你可以按上面的问题回答，也可以自由描述。我会根据你的回答自动匹配技术栈和流程级别。
```

### 输入模板（可选）

```markdown
## 需求
[一句话描述要什么]

## 背景
[为什么需要？]

## 约束
- [不能改的东西]
- [必须兼容的东西]

## 方案偏好（可选）
[如果有倾向性方案就写，没有就让 AI 自己分析]
```

## 需求分级触发

### AI 自动判定需求级别

```
用户需求
    │
    ▼
┌─────────────────────────────────────────────────────┐
│  AI 判定：                                           │
│  · 仅涉及 ≤1 文件且改动 ≤20 行？ → L1 极轻            │
│  · 新增API / 修Bug / 单模块功能 / 涉及 2-5 文件？ → L2 标准  │
│  · 重构 / 跨模块 / 新系统 / 安全相关 / >5 文件？ → L3 重型  │
└─────────────────────────────────────────────────────┘
    │
    ▼
AI 告知：「我判定这个需求为 L2（标准），将产出 spec + verify 两个文档。
          如需完整流程请告诉我。」
```

### 判定矩阵

```
                    │ 1个文件     │ 2-5个文件   │ 5+个文件
────────────────────┼─────────────┼─────────────┼───────────
纯表面变更（改名/改值）│ L1          │ L1          │ L2
功能增强 / 修Bug      │ L2          │ L2          │ L3
新增子系统 / 模块      │ L2          │ L3          │ L3
架构级重构            │ —           │ L3          │ L3
跨技术栈              │ L2          │ L3          │ L3
安全相关（认证/加密）   │ L3          │ L3          │ L3
数据迁移（DB schema等）│ L2          │ L3          │ L3
性能优化（算法级改动）  │ L2          │ L2          │ L3
依赖升级（大版本）      │ L2          │ L2          │ L3
```

### 用户可覆盖

用户回复「L3」或「走完整流程」→ 升级为 L3；回复「直接改」或「快速」→ 降级为 L1。

## 并发需求管理

### specs/INDEX.md

每个项目维护一个 `specs/INDEX.md`，格式见 `conventions/naming-and-formats.md`。

- 每个新需求自动在 INDEX.md 登记
- 同时只有一个需求处于 `🔨 implementing`
- L1 需求可插队执行，不阻塞当前 L2/L3
- L3 过程中收到 L2 需求 → 完成后从 DevLog 恢复 L3 进度

---

## L1 极轻通道 — 直接执行

```
用户确认改动内容 → AI 直接改代码 → git diff review
```

**规则**：
- 不改动时只说一句话描述改动
- 不产出任何 Markdown 文件
- 改完后展示 git diff 供用户确认
- 结束后可选记录 DevLog（学到新东西或踩坑时）

---

## L2 标准通道 — 简化 3 阶段

```
📋 [1/3] 方案设计 → 🔨 [2/3] 执行实现 → ✅ [3/3] 验证 + DevLog
```

### 阶段 L2-1：方案设计

产出 `specs/L2/{feature-name}/design.md`。格式见 `templates/spec-L2.md`。

**规则**：一个文件搞定，用户审核通过后进入 L2-2。

### 阶段 L2-2：执行实现

AI 按 design.md 直接写代码。写完后自动运行：编译 / lint / typecheck / 测试（如果有）。

### 阶段 L2-3：验证 + DevLog

产出两个文件：
- `specs/L2/{feature-name}/verify.md`（格式见 `templates/verify.md`）
- `Aegis/rules/DevLogs/{项目名}_P{阶段号}_{功能描述}_{YYYY-MM-DD}.md`

---

## L3 重型通道 — 完整流程

```
📋 [1/7] 头脑风暴 → 📋 [2/7] 提案 → 📐 [3/7] 技术设计
→ 📝 [4/7] 需求规格 → 📋 [5/7] 任务清单 → 📋 [6/7] 集成审核
→ 🔨 [7/7] 执行验证 → ✅ 收尾仪式 + DevLog
```

### 阶段 L3-1：头脑风暴

产出 `specs/L3/{feature-name}/01-brainstorm.md`。格式见 `templates/brainstorm.md`。

### 阶段 L3-2：提案

产出 `specs/L3/{feature-name}/02-proposal.md`。格式见 `templates/proposal.md`。

**存量代码协作**：如果是在已有项目中加功能（非从零开始），提案前增加一步：
- AI 先分析现有架构：读取关键文件 → 绘制模块依赖图 → 列出可复用组件
- 在提案中追加「现有架构分析」一节

### 阶段 L3-3：技术设计

产出 `specs/L3/{feature-name}/03-design.md`。格式见 `templates/design.md`。

### 阶段 L3-4：需求规格

产出 `specs/L3/{feature-name}/04-spec.md`。格式见 `templates/spec-L3.md`。

**00-MetaSpec（可选）**：如果需求是 L3 重构类，可在阶段 1 之前产出 `00-{MetaSpec}.md`。格式见 `templates/meta-spec.md`。
00 是「宪法」— 跨批次约束；04 是「法律」— 当前批次具体需求。04 不能违反 00。

### 阶段 L3-5：任务清单

产出 `specs/L3/{feature-name}/05-tasks.md`。格式见 `templates/tasks.md`。

### 阶段 L3-6：集成审核

产出 `specs/L3/{feature-name}/06-review.md`。格式见 `templates/review.md`。

**审查悖论处理**：
- AI 子代理自行审查，输出双版本：
  a) **专业版**：技术细节、API 签名审查、数据流验证
  b) **口语版**：用通俗语言解释「这个设计在做什么、为什么这样选、有什么风险、替代方案是什么」
- 每个关键判断附带**可验证的参考来源**（官方文档链接、GitHub 讨论、权威博客）
- 用户只需审查口语版 + 参考链接，自行验证或学习
- 审查可能有误 → 这是可接受的代价，DevLog 中记录修正过程

### 阶段 L3-7：执行验证

1. AI 按 `05-tasks.md` 逐项执行代码修改
2. AI 自动运行：lint → typecheck → build → test
3. 产出 `07-verify.md`（格式见 `templates/verify.md`）

### 迭代退回机制

L3 不是瀑布。任何阶段发现问题都可以退回到之前的阶段：

```
用户：「回到阶段 N」或「{阶段N的文档} 有问题」
    │
    ├── 保留阶段 1 ~ (N-1) 的文档不变
    ├── 从阶段 N 重新生成
    ├── 阶段 (N+1) ~ 当前 的后续文档自动标记废弃（文件名加 .deprecated）
    └── 继续推进
```

AI 在每阶段结束后主动问一句：「这部分没问题的话我们进入下一阶段？」

## 收尾仪式（L3 Close-out）

L3 需求完成后，AI 必须执行以下收尾步骤（不跳过）：

```
□ 1. 写 DevLog（详细开发日志）
     → 失败处理：如果忘记写，下次对话从 DevLog 缺失可发现，补写

□ 2. 更新 specs/INDEX.md 状态为 ✅ done
     → 失败处理：AI 在下次对话开始时检查 INDEX.md 是否有未关闭的需求

□ 3. TempData 清理：检查是否有可提炼到 TechStack 的资料
     → 满足提炼标准？合并到 TechStack
     → 不满足？保留链接摘要，删除详细内容
     → 失败处理：季度归档时统一清理；提炼是增值操作，错过不致命

□ 4. 5 维度审视（主动检查工作流是否需要改进）：
     □ 目录结构是否合理？
     □ 文档职责边界是否清晰？
     □ 流程是否冗余或缺失？
     □ 是否有 AI 容易误解的指令？
     □ 用户是否在某些环节有认知负担？
     → 发现问题后在对话中提出，征得用户同意后改进
     → 失败处理：下次需求完成时补做；审视是持续改进的「保险」，偶尔漏一次不影响

□ 5. 问用户：「有没有值得沉淀到 TechStack 或规范中的经验？」
     → 失败处理：用户自己会提，不问也不会丢
```

**兜底机制**：如果整个收尾仪式被跳过（用户关 IDE / 对话意外终止），AI 在下次对话开始时：
1. 从 DevLog 发现 L3 已执行但无「收尾仪式」记录
2. 补充执行收尾步骤 2~5

## 开发日志规范

> 完整命名格式和日志模板见 `conventions/naming-and-formats.md`。此处只定义流程规则。

### 日志格式

```markdown
# {项目名} — 开发日志

**日期**: YYYY-MM-DD HH:MM
**阶段**: 阶段{两位数字} — {简短描述}
**技术栈**: {Unity / Python / TypeScript / ...}
**需求级别**: L1 / L2 / L3
**当前进度**: {阶段名} / {状态}
**下一动作**: {用户审核 / AI等待指令 / 进入阶段N}

---

## 需求背景
[为什么需要做这个功能]

## 问题原因
[Bug 根因分析 / 技术方案选择理由]

## 解决过程

### 步骤 N: {做了什么}
- 修改文件: `path/to/file`
- 关键决策: [为什么这样选，不选替代方案]
- 遇到的问题: [如果有]
- 当前状态: [✅完成 / 🔨进行中 / ⏸️暂停]

## 参考资源
- [文章标题](链接) — 博客/官方文档
- [GitHub Issue #xxx](链接)

## 经验教训
[学到什么、下次可以怎么做得更好]

## 产出文件清单
| 文件 | 操作 |
|------|------|
```

**进程恢复关键字段**：
- `当前进度` + `下一动作`：AI 读取后能精确知道「做到哪了、下一步做什么」
- `步骤 N.当前状态`：标记每个步骤是否完成，支持断点续传
- `关键决策`：记录「为什么这样选」防止 AI 恢复后重复讨论已决定的问题

**时间戳规则**：日期精确到天，时间精确到分钟（HH:MM），确保进程恢复时能判断是最新的。

### 日志记录时机
- L1：可选（学到新东西或踩坑时）
- L2：结束后必须记录
- L3：每个批次完成后追加，结束时提交完整日志

| 维度 | 00-MetaSpec | 04-spec |

| 冲突时 | 以 00 为准 | — |

## AI 行为准则

### 交互协议

1. **需求判定**：先判断 L1/L2/L3，告知用户，沉默 = 确认
2. **生成文档后暂停**，等待用户审核
3. **审核未通过不能进入下一阶段**
4. **疑问优先提问**，不猜测用户意图
5. **回退机制**：用户说「回退到阶段 N」→ 保留 1~(N-1) 文档不变，从 N 重新生成，后续文档加 `.deprecated`
6. **进度可见**：每阶段开头带 `📋/📐/📝/🔨/✅` 标示
7. **收尾必做**：L3 结束时执行完整收尾仪式
8. **进程恢复**：每次对话开始先读最新 DevLog 恢复上下文
9. **并发管理**：新需求自动登记 INDEX.md，同时只有 1 个需求在 `🔨 implementing`
10. **存量分析**：在已有项目中加功能时，先分析现有架构再提案

### 审查输出规范

AI 子代理审查时输出双版本：
- **专业版**：完整技术细节
- **口语版**：通俗解释 + 参考链接
- 用户只审口语版即可

### 禁止行为

- ❌ 跳过审核直接执行（L3 流程）
- ❌ 在提案/设计阶段写实现代码
- ❌ 自行修改已审核通过的文档
- ❌ 技术栈文件不存在时直接猜测，必须走搜索+审查流程
- ❌ 使用 TempData 中未经审查代理验证的资料
- ❌ 忘记写 DevLog（L2/L3 必须）
- ❌ 忘记更新 specs/INDEX.md
- ❌ 忘记执行收尾仪式（L3）

## 持续改进与维护

### 主动性要求

每个需求完成后，AI 从以下 5 维度做主动审视：

| 审视维度 | 具体问题 |
|----------|----------|
| 目录结构 | specs/ 下是否有不该放一起的文件？命名一致吗？ |
| 文档职责边界 | 有没有两份文档写同一件事？有没有该写的没写？ |
| 流程冗余/缺失 | 有没有某阶段产出从不被读？缺必要的决策点？ |
| AI 误解风险 | 有没有指令可能被 AI 误读？ |
| 用户认知负担 | 有没有文档用户明显不想看但被迫审？ |

发现问题后：对话中讨论 → 用户同意 → 更新对应文件 → DevLog 记录改动原因。

### 自举能力

工作流自身走 L3 升级自己：

```
用户：「升级 dev-workflow 规则」→ AI 走 L3：
  specs/L3/dev-workflow-v{N}/
    ├── 00-Constraints.md
    ├── 01-brainstorm.md
    ├── 02-proposal.md
    ├── 03-design.md
    ├── 04-spec.md
    ├── 05-tasks.md
    ├── 06-review.md
    └── 07-verify.md
```

### 归档策略

- **季度归档**：每季度创建 `archive/`，移入三个月前的 DevLog
- **精华提炼**：从多篇 DevLog 中提炼教训清单追加到 TechStack 末尾
- **TempData 清理**：功能完成后 7 天内清理不需要的；季度清理 90+ 天未引用的

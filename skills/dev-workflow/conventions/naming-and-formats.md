# 命名与格式约定

> **单一事实来源** — DevLogs、TempData、specs 目录的命名和格式规范。
> SKILL.md、Aegis.md、各 README 中不再重复定义，统一引用本文。

---

## DevLogs

### 文件命名

```
Aegis/rules/DevLogs/{项目名}_P{两位数字}_{功能描述}_{YYYY-MM-DD}.md
```

示例：
- `My_Fish_01_P01_ProjectRefactor_2026-05-30.md`
- `Aegis_P02_WorkflowOptimization_2026-05-30.md`

### 日志格式

见 [SKILL.md 开发日志规范](#)。日志必须包含以下进程恢复关键字段：

| 字段 | 用途 |
|------|------|
| `当前进度` | AI 恢复时知道做到哪了 |
| `下一动作` | AI 知道下一步做什么（用户审核 / 进入阶段N） |
| `步骤 N.当前状态` | 标记 `✅完成 / 🔨进行中 / ⏸️暂停` |
| `关键决策` | 防止 AI 恢复后重复讨论已决定的问题 |

### 日志记录时机

- **L1 任务**：结束后可选记录（学到新东西或踩了坑时）
- **L2 任务**：结束后必须记录
- **L3 任务**：每个批次完成后追加，最终结束时提交完整日志

### 进程恢复

AI 在每次对话开始时：
1. 读取最新日期的 DevLog
2. 检查 `当前进度` + `下一动作` + `步骤 N.当前状态` 恢复上下文
3. 信息不足时，依序读取上一份
4. 告知用户当前进度

## TempData

### 文件命名

```
Aegis/rules/TempData/{主题}_{YYYY-MM-DD}.md
```

示例：
- `unity_addressables_memory_leak_2026-05-30.md`
- `python_asyncio_best_practices_2026-05-30.md`

### 资料结构

```markdown
# {主题}

## 原始来源
- [文章标题](链接) — 一句话摘要
- [GitHub Issue #xxx](链接) — 一句话摘要

## 审查结论
- ✅ 可复用 — {理由}
- ⚠️ 有条件可用 — {条件}
- ❌ 不可用 — {原因}

## 关键要点
- {从资料中提取的通用知识}
```

### 提炼到 TechStack 的标准

功能开发过程中，如果 AI 发现 TechStack 缺少某块技术知识（如「对象池」「协程」），流程如下：

```
1. AI 网上搜索 → 审查代理验证 → 暂存 TempData
2. 使用该知识完成当前功能
3. 功能完成后提炼：将核心要点精简写入 TechStack
```

**提炼原则**：
- 写入 TechStack 的内容必须是**广泛通用的知识**（如「对象池模式」），不是具体实现（如「子弹对象池」）
- 保持 TechStack 精简——提炼后不超过原文件的 30%
- 如果该主题已经有足够广泛的第三方文档/官方文档，TechStack 只需记录「概念 + 参考链接 + 常见陷阱」，不展开

**不需要提炼的情况**：
- 该知识仅在当前项目特定场景有用（一次性的）
- 网上已有更权威的完整文档，Aegis 只需引用链接即可

### 清理规则

- 功能完成后 7 天内：确认不再需要的 TempData 文件 → 删除
- 已提炼到 TechStack 的：原 TempData 文件可保留链接摘要（1-2 行），删除详细内容
- 每季度清理一次超过 90 天未引用的 TempData

## specs/INDEX.md

### 格式

```markdown
# 需求索引

| ID | 需求名 | 级别 | 状态 | 开始日期 | 最后活动 |
|----|--------|------|------|----------|----------|
| REQ-001 | xxx | L3 | 📋 proposal完成 | 2026-05-30 | 2026-05-30 |
```

### 状态值

| 状态 | 含义 |
|------|------|
| 📋 brainstorm | 方案讨论中 |
| 📋 proposal | 方案已定，待审核 |
| 📐 design | 技术设计中 |
| 📝 spec | 需求规格编写中 |
| 📋 review | 审核中 |
| 🔨 implementing | 代码实现中 |
| ✅ done | 已完成 |
| ⏸️ paused | 暂停 |
| ❌ cancelled | 取消 |

### 并发规则

- 同时只有一个需求处于 `🔨 implementing`
- L1 需求可插队执行，不阻塞当前 L2/L3
- L3 过程中收到 L2 需求：完成后从 DevLog 恢复 L3 进度

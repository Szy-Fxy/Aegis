```markdown
# 技术设计

## 新 API
```{语言}
{完整签名：类型 + 参数名 + 默认值 + 返回值}
```

## 修改 API
```{语言}
{修改前} → {修改后}
```

## 删除
| 删除项 | 替代方案 |
|--------|----------|

## 保留
| 保留项 | 原因 |
|--------|------|
| {明确保留的类/方法} | {为什么不动它} |

## 数据流
```
输入 → 处理步骤 1 → 处理步骤 2 → 输出
```

## 依赖关系
```
{模块 A} ──→ {接口 I} ──→ {模块 B}
```

## 错误处理策略
- [正常路径错误处理方式]
- [异常路径降级方案]
- [用户可见的错误信息设计]

## 安全考量
- [涉及的用户输入及校验方式]
- [涉及的敏感数据及保护措施]
- [新增 API/端点是否需要认证]

## 测试策略
- [单元测试覆盖范围]
- [集成测试场景]
- [边界条件测试用例]
```

**规则**：
- 「保留」列表是防御性的，防止 AI 误删
- 数据流图中用文字描述即可，不需要 PlantUML
- 依赖关系标注接口/抽象，不是具体类
- 错误处理策略必须覆盖正常路径和异常路径
- 安全考量涉及用户数据时必须详细说明
- 测试策略至少列出 3 个关键测试场景

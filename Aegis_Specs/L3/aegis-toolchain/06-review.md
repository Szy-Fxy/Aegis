## 必须立即修正的 P0 问题（3项）

### P0-1: proposal 验收标准 #1 与风险边界自相矛盾

**发现者**: 唱反调 + UX  
**问题**: proposal 承诺"自动注入"，但风险边界自己承认"Hana 不支持用户消息前置钩子"。用户验收时永远无法通过这条标准。  

**修正**: 将验收标准 #1 改为诚实表述，并拆分为两个阶段目标。

### P0-2: `aegis start` 参数语义矛盾

**发现者**: 纠错  
**问题**: proposal 中 `aegis start REQ-003`（参数是 ID），design 中 `start(title: str)`（参数是标题名称）。API 契约级别矛盾。  

**修正**: 统一为 title（需求名称），自动生成 REQ-ID。

### P0-3: 无安装指南 / 用户无法5分钟跑通

**发现者**: UX  
**问题**: 文档链缺少 Quick Start，用户不知道如何安装、首次使用。  

**修正**: 在 02-proposal 末尾增加 Quick Start 章节。

## 需要在进入实现前解决的 P1 问题（4项）

### P1-1: CLI 全部依赖 AI 自律，只有 Git Hook 有强制力

**发现者**: 唱反调（核心矛盾 #2）  
**问题**: CLI 的 start/check/advance/devlog 全部需要 AI 主动调用，本质上和"AI 读 SKILL.md"没有区别。  

**决策**: **接纳批评，但不改变 Phase 1 范围。**  
Phase 1 的 CLI 定位不是"强制"，而是"便利 + 精确"。CLI 解决了状态不可靠（state.json 结构化）和 BOUNDARY CHECK 不可跳过（check 由脚本执行而非 AI 自查）的问题。真正强制交给 pre-commit hook。  

**行动**: 在文档中明确区分 CLI（便利+精确）和 Hook（强制），不再宣称 CLI 是"强制"机制。

### P1-2: 三套状态存储冗余

**发现者**: 唱反调（核心矛盾 #3）  
**问题**: state.json + INDEX.md + DevLog 三个数据源维护同一份生命周期信息。  

**决策**: **短期并存，中期收敛。**  
Phase 1 保留三者：state.json 为结构化权威源，INDEX.md 为人类可读速查，DevLog 为文本记录。`aegis check` 强制校验三者一致性。Phase 2 考虑将 INDEX.md 改为 state.json 的自动生成快照。  

### P1-3: StateCorruptedError 暴露 Python 内部类型名

**发现者**: UX（P0 #4）  
**问题**: 非技术用户看到异常类名会恐慌。  

**修正**: 错误提示改用自然语言 + 修复建议，不暴露类名。

### P1-4: 两套验收标准不统一

**发现者**: UX  
**问题**: proposal 6条 vs spec 8条，互不包含，用户不知道以哪个为准。  

**修正**: 统一为 proposal 中的 6 条（用户语言），spec 中的技术标准挪入 tasks 验收条件。

## 从唱反调核心批评中采纳的改进

唱反调子代理的三个核心矛盾虽然尖锐，但有一半（矛盾 #2/#3）我不完全认同，以下是我的回应：

| 唱反调批评 | 我的回应 | 行动 |
|-----------|---------|------|
| "CLI 把 AI 自律换了个马甲" | **部分认同**。CLI 确实不强制调用，但 CLI 提供了 AI 自律做不到的事：结构化状态（不会再写错 phase）、独立 BOUNDARY CHECK（不由 AI 自查）、精确断点恢复。 | 文档中不再宣称 CLI "强制"，定位改为"精确 + 便利" |
| "只有 Git Hook 有强制力，为什么不做精简版" | **不认同**。精简版（纯 pre-commit + state.json）只能阻断 commit，不能解决"AI 不知道自己该做什么"的问题。CLI 是 AI 的工作辅助工具，Hook 是用户的兜底防线。两者职责不同。 | 维持 Phase 1 范围，但在文档中对 CLI 定位做诚实说明 |
| "三套存储冗余" | **部分认同**。短期并存是过渡期的代价。Phase 2 会统一。 | 在 check 中加入一致性校验 |

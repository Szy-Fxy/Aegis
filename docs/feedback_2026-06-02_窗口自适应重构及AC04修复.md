## 五、改进建议（AI 视角）

### 基于本次任务，Aegis 规则哪里需要改进？

1. **方案验证的前置检查**：对于涉及"缩放/自适应"等 CSS 布局的方案，应该在 L2-2 前增加一个"快速原型验证"步骤——让 AI 在浏览器/Electron 中实际观察效果，而不是仅凭 CSS 知识推理。当前流程中没有"实现前验证方案"的环节。

2. **验收标准的用户视角对齐**：AC-01"不断层""无溢出"是技术视角的描述，用户实际上关心的是"窗口拉大时内容变多大"。建议在 L3-2 提案阶段加入用户场景模拟："用户拖大窗口时他看到什么"'。

3. **方案转向的回退机制**：当用户验收失败时，当前的流程是"退回重来"但缺乏明确的"原文档标记 + 新方案路径"的引导。L3 的迭代退回机制（`.deprecated` 标记）在本次没有被使用，因为从三层容器到 flex 是方案层面的转向，不完全是 L3 内部的迭代退回。

### 本次任务中，哪些规则特别有效？哪些完全没用？

- **特别有效的规则**：BOUNDARY CHECK（read 确认文件存在）在 L3 前期阶段有效保证了文档完整性。子代理审查的备选方案调研有价值（确认三层容器避开 Chromium bug）。

- **完全没用的规则**：验收标准写进 proposal 并同步到 verify 的做法，在本次任务中失效了——因为 AC-01 的验收标准（"无溢出、无裁剪"）本质上是错误的验收方向，用户真正在意的不是溢出而是填充。验收标准写得再精确，方向错了就没用。

# AI 开发治理与安全审查 — 最佳实践汇总

## 原始来源
- [GitHub Copilot Coding Agent 安全最佳实践](https://github.github.io/awesome-copilot/learning-hub/using-copilot-coding-agent/) — 安全考虑、选择合适任务、设置成功环境
- [Agentic Coding 安全原则](https://agentic-coding.github.io/) — 不要将敏感信息输入外部 AI、警惕数据泄漏
- [Spec-Driven Development 企业级实践](https://www.resultsense.com/insights/2025-09-12-spec-driven-ai-development/) — 将安全、合规、设计系统从隐性知识变为可执行输入
- [SDD 最佳实践](https://arceapps.com/blog/spec-driven-development-ai/) — 生产级 Spec 的 7 个 Section、Generator-Critic 模式

## 审查结论
- ✅ **可复用 — 安全与合规作为 Spec 的第一等公民**：安全、设计系统、合规要求应在 spec 阶段就嵌入，而非事后审查
- ✅ **可复用 — Generator-Critic 模式**：代码生成与验证分离，减少 AI 幻觉
- ✅ **可复用 — 决策卫生（Decision Hygiene）**：质量计划需要显式的标准、约束和非功能需求
- ⚠️ **有条件可用 — 组织约束作为可执行输入**：安全策略、合规要求从"部落知识"转为 AI 可读取的规则文件
- ❌ **暂不可用 — 自动化合规检查**：需要 CI/CD 集成，超出 Aegis 当前纯 Markdown 范围

## 关键要点

### 1. 安全与合规 — 必须在 Spec 阶段嵌入

**核心原则**：安全、设计系统、合规要求属于 spec 的第一等公民，不应在事后审查中才关注。

**具体实践**：
- 在 spec 模板中明确包含「安全要求」和「合规约束」section
- 在 global.md 中定义不可协商的安全规则
- 子代理审查中增加「安全审查」维度

**Aegis 当前差距**：
- global.md 缺少安全相关约束
- spec 模板（spec-L3.md）没有安全要求 section
- 子代理审查未覆盖安全检查

### 2. Generator-Critic 模式 — 代码生成与验证分离

**核心思想**：将代码生成（Generator）和代码验证（Critic）分离为两个独立的 AI 调用或子代理。

**为什么有效**：
- AI 在生成代码时倾向于"自信地犯错"
- 同一个 AI 难以同时切换"生成模式"和"审查模式"
- 分离后，Critic 可以用不同的提示词和更严格的检查标准

**实践方式**：
1. Generator 生成代码 → 输出到文件
2. Critic 审查代码 → 检查：功能正确性、安全漏洞、代码风格、测试覆盖
3. 发现问题 → Generator 修复 → Critic 重新审查
4. 循环直到 Critic 通过

**对 Aegis 的启发**：Aegis 的子代理审查已经部分实现了这个模式，但可以在 06-implement 阶段更明确地分离生成和验证。

### 3. 决策卫生（Decision Hygiene）

**核心原则**：质量计划需要显式的标准、约束和非功能需求——AI 只能尊重被明确表达的内容。

**四类必须显式表达的约束**：
1. **技术标准**：编码规范、架构模式、测试策略
2. **安全约束**：认证方式、数据加密、PII 处理
3. **合规要求**：行业法规、数据驻留、审计日志
4. **非功能需求**：性能指标、可扩展性、可维护性

**反模式**：假设 AI "应该知道"这些约束 → AI 不会自动遵守未明确表达的约束

### 4. 生产级 Spec 的 7 个 Section

基于 SDD 最佳实践，一个完整的 spec 应包含：

1. **意图与背景**：为什么做、解决什么业务问题
2. **用户故事与验收标准**：谁、做什么、为什么、验收条件
3. **功能需求**：编号的功能需求（FR-001, FR-002...）
4. **API 契约**：端点、请求/响应格式、状态码、错误格式
5. **架构设计**：组件关系、数据流、技术决策
6. **边界条件与错误处理**：Edge cases、异常路径、降级策略
7. **非功能需求**：性能、安全、可扩展性、可维护性

**对 Aegis 的启发**：Aegis 的 spec-L3.md 模板（8 段式）已覆盖大部分，但缺少「API 契约」和「非功能需求」的独立 section。

### 5. AI 编码代理的安全最佳实践

**选择合适任务**：
- ✅ 适合：CRUD 操作、模板代码、单元测试、重构、文档
- ⚠️ 谨慎：复杂业务逻辑、安全关键代码、性能敏感代码
- ❌ 不适合：加密算法实现、认证系统核心逻辑

**安全考虑**：
- 永远不要让 AI 代理访问生产环境
- 限制 AI 代理的文件系统访问范围
- 审查 AI 生成的所有依赖和库引用
- AI 生成的代码必须通过安全扫描

**设置成功环境**：
- 为代理提供清晰的编码标准
- 定义明确的边界（哪些文件不能碰）
- 保持 PR 小而可审查（<100 行变更）
- 始终运行测试套件验证

### 6. 对 Aegis 的安全与治理增强建议

| 增强项 | 具体措施 | 优先级 |
|---|---|---|
| global.md 安全规则 | 添加安全约束 section：敏感数据、认证、加密 | 高 |
| spec 模板安全 section | 在 spec-L3.md 中增加「安全要求」和「非功能需求」 | 高 |
| 子代理安全审查 | 新增 security-auditor 子代理角色 | 中 |
| Generator-Critic 分离 | 在 06-implement 中明确分离生成和验证步骤 | 中 |
| 组织约束规则文件 | 创建 `rules/security.md` 和 `rules/compliance.md` | 低 |
| CI/CD 安全检查 | 集成 linting、SAST 到验证流程 | 低（远期） |

### 7. PR 审查最佳实践

**审查 AI 生成的 PR 时**：
1. 逐行审查，不要假设正确性
2. 检查是否符合架构决策
3. 验证测试覆盖了边界条件
4. 确认没有引入新的依赖
5. 检查安全漏洞（注入、XSS、敏感数据暴露）
6. 确认代码风格一致

**迭代方法**：通过 PR 评论驱动 AI 修正，而非手动修改 AI 生成的代码。

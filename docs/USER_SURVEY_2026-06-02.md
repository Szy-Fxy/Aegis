## 五、改进建议（AI 视角）

### 基于本次任务，Aegis 规则哪里需要改进？

1. **L1 fix 也应有流程门槛**：不能因为是"小修"就不走流程。至少需要：用户确认问题→定级→告知要改→改后展示→更新 INDEX+DevLog
2. **"已验收代码"的修改权限**：对用户明确说"已验收通过"的代码，AI 不得以任何理由擅自修改。即使认为有改进空间，也只能提意见，不能动手
3. **状态机和朝向等视觉敏感代码**：涉及视觉效果的修改，每次改动后都应该让用户验证，不应连续改多轮再一次性验证

### 本次任务中，哪些规则特别有效？哪些完全没用？

- **特别有效的规则**：
  - L3 的 Brainstorm → Design → Spec 流水线有效，方案阶段充分讨论避免了更大的返工
  - BOUNDARY CHECK 在 L3 阶段有用（确认文档存在再进入下一步）
  
- **完全没用的规则**：
  - L1 的"展示 git diff"——在 HanaAgent 环境中没有 git，这条规则无法执行
  - "等待用户确认"——写了但没做到，AI 在压力下容易跳过此步
